READ THE KEYBOARD 1

The Missing Half

Every program you've written so far does the same thing every time you run it. It doesn't matter who runs it or when — the output is fixed. That's useful for demonstrations, but it's not interactive.

The missing half is input. To be truly useful, a program needs to wait for the person at the keyboard, read what they type, and respond accordingly. That's what KEY does.

KEY — Wait for a Keypress

Illustration A large key-shaped button hovering above the data stack. When pressed, it drops a number (the character code) onto the top of the stack. The number is labeled with both the key symbol (Y) and its ASCII value (89). Minimal, iconic.

KEY  ( -- c )

Wait until a key is pressed. Push its ASCII character code onto the stack. Does not echo the character.

KEY blocks — it does nothing until a key is pressed. Your program simply pauses at that point, waiting. When a key arrives, its ASCII value lands on the stack and execution continues.

The simplest use: read a key and immediately display it. 

KEY EMIT

Press any key — that character appears on screen. The round-trip is: KEY pushes the code, EMIT consumes and displays it.

What KEY Returns

KEY pushes a standard ASCII value: letters are uppercase (the CoCo keyboard produces uppercase by default), digits are their ASCII codes, and a few special keys have well-known values:

Key ASCII Decimal
A – Z $41–$5A65–90
0 – 9 $30–$3948–57
Space $20 32
Enter $0D 13

Because these are just numbers, you can compare them with =, <, > — all the comparison words from Chapter 7 apply directly.

The CoCo keyboard returns uppercase letters even without Shift held down. CHAR Y compiles to 89 ($59). That's exactly what KEY pushes when you press Y. No case conversion needed.

READ THE KEYBOARD 2

Stack Trace: KEY CHAR Y = IF … ELSE … THEN

Suppose the user presses Y. Here's what happens inside a word that contains KEY CHAR Y = IF .YES ELSE .NO THEN:

Step Data stack
KEY → user presses Y ( 89 )
CHAR Y → literal 89 ( 89 89 )
= → equal, push -1 ( -1 )
IF → non-zero, take true( )
.YES → prints YES ( )
THEN → continues ( )

If the user presses N instead: = produces 0, IF skips to ELSE, and .NO runs. Same structure as Chapter 7 — the only new ingredient is KEY at the top.

YORN — Yes or No

A reusable yes/no prompt: 

: YORN  ( -- )
  KEY CHAR Y = IF .YES ELSE .NO THEN CR ;

Call YORN and the program waits. Press Y and it prints YES. Press anything else and it prints NO. One call per decision.

KEY does not echo the character — the keypress is silent until your program decides what to do with it. If you want to show what was typed, add DUP EMIT right after KEY before you use the value for anything else.

Reading a Key Without Waiting: KEY?

KEY always blocks. If you need to check whether a key is available without stopping the program, standard Forth provides KEY? — but for now KEY is enough: every interesting program in these chapters has a clear moment when it wants input, and blocking is exactly right.

Chapter Program

\ Chapter 8 example — Read the Keyboard

: .YES   CHAR Y EMIT CHAR E EMIT CHAR S EMIT ;
: .NO    CHAR N EMIT CHAR O EMIT ;

: YORN  ( -- )
  KEY CHAR Y = IF .YES ELSE .NO THEN CR ;

: MAIN
  YORN
  YORN
  YORN ;

MAIN HALT

The program waits three times. Each time you press a key it replies YES (if you pressed Y) or NO (anything else).

Forth Words

KEY
( -- c ) wait for keypress, push ASCII value

Patterns

KEY CHAR X = IF … THEN to act on a specific key

KEY EMIT to echo a keypress

KEY DUP EMIT … to echo and then test

Concepts

KEY blocks until a key is pressed

CoCo keyboard returns uppercase letters

Enter = 13, Space = 32

Key codes are just numbers — all comparison words apply
