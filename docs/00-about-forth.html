ABOUT FORTH

Where Forth Came From

Forth was invented in the late 1960s by Charles Moore, a programmer at the National Radio Astronomy Observatory. He needed a language small enough to run on the modest computers of the day, expressive enough to talk directly to telescopes, and fast enough to be useful in real time. He built it himself, and named it FOURTH — meaning fourth-generation language. The operating system of the time limited filenames to five characters. It became FORTH.

The language spread quickly through the 1970s because it solved a real problem: it was tiny, fast, and could be made to run on almost any hardware in a matter of days. FORTH Inc. commercialised it in 1976, and implementations appeared for the Intel 8080, Motorola 6800, and Zilog Z80. By the early 1980s, there were Forth systems for nearly every microcomputer that existed — including the CoCo.

Electronic Arts shipped several games written in Forth in the 1980s, including Starflight (1986) and Adventure Construction Set (1984). Forth was everywhere.

Forth's Unusual Philosophy

Most languages compile your source code into a binary, and that binary runs on the machine. Forth is different: the compiler, the editor, the runtime, and your program all live together in the same environment, on the target machine. You type a word definition at a prompt, and Forth compiles it immediately — into the same dictionary where all the other words live. The system extends itself.

This philosophy — the language compiling itself, on the machine you're programming — is why Forth has a devoted following fifty years later. It's intimate in a way that most programming environments are not. You are always one step from the hardware.

Forth in the Real World

Forth has turned up in some surprising places:

  • OpenFirmware — an IEEE standard boot environment written in Forth, used as the boot ROM in Apple PowerPC Macs, Sun workstations, IBM POWER machines, and the OLPC XO laptop. Every time a PowerPC Mac started up, Forth ran first.
  • Philae — the ESA comet lander that touched down on Comet 67P in 2014 ran Forth in its onboard systems.
  • Embedded systems — Forth remains genuinely used in aerospace, industrial control, and safety-critical applications where determinism and tiny footprint matter.
  • Today — GForth (GNU Forth) is actively maintained. Mecrisp is popular on ARM Cortex-M microcontrollers. The retro computing community runs Forth on 8-bit machines for the same reasons Moore did in 1969: it fits, it's fast, and it's close to the metal.

How This Forth Is Different

Traditional Forth is self-hosted: the compiler lives on the target machine, the dictionary lives in RAM, and you interact with the system directly. That's elegant, but it requires RAM for the dictionary, ROM or disk for the compiler, and some kind of input loop to drive it all. On a stock CoCo, that means a disk drive, an OS, and a Forth system large enough to include its own editor and compiler.

This project takes a different approach. Instead of running the Forth compiler on the CoCo, we run it on a modern machine — your Mac or Linux computer. You write Forth source in a plain text file. The cross-compiler (fc.py) compiles it into threaded bytecode: a compact binary where every word reference is a 2-byte address pointing into the kernel's dispatch table. That binary is loaded onto the CoCo, where a small kernel — roughly 100 bytes of 6809 assembly — executes it.

The Key Distinction

Traditional Forth: compiler on the target, dictionary in RAM, interactive at the prompt.

This Forth: compiler on the host, no dictionary on the target, pure execution only.

The CoCo never sees source text. It only ever executes the compiled result.

This is sometimes called token-threaded Forth in the literature. Word references in the compiled binary are tokens — fixed addresses in a kernel dispatch table — rather than self-describing dictionary entries. It's closer in spirit to how a JVM or Lua bytecode VM works than to how traditional Forth works.

The trade-off is deliberate. You lose the interactive REPL and the self-extending dictionary. You gain a system that needs no OS, no disk, and no RAM beyond what your program actually uses. The entire runtime fits on a ROM cartridge. Plug it in, load your program, run it.

Why the 6809?

The Motorola 6809 is an unusually good fit for Forth. Traditional Forth needs two stacks: a data stack for working values, and a return stack for tracking word calls. Most processors have one hardware stack and have to fake the second in software.

The 6809 has two hardware stack pointers: S and U. They map directly onto Forth's return stack and data stack. Nothing is faked. The 6809 was practically designed for this, which is why Forth runs on it so naturally — and why the kernel is so small.
