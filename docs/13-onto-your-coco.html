GETTING IT ONTO YOUR COCO 1

From Cross-Compiled Binary to Real Hardware

Every program in this book was cross-compiled on your modern computer and tested in XRoar. The output is a standard DECB .bin file — the same format the CoCo's BASIC uses for CLOADM. Getting it to real hardware means copying that file somewhere the CoCo can read it.

There are three practical paths, depending on what hardware you have.

Path 1 — CoCoSDC (Recommended)

The CoCoSDC is a cartridge that gives the CoCo a real SD card slot. It presents the SD card as a standard disk drive to BASIC.

  1. Copy your .bin file to a FAT-formatted SD card on your modern computer.
  2. Insert the SD card in the CoCoSDC.
  3. At the BASIC prompt, type: 
    CLOADM"GUESS.BIN"
    BASIC loads both the kernel block and the app block, then executes at the kernel's START address.

The .bin file contains two data blocks (kernel at $1000, app at $2000) and an end block with the exec address. CLOADM loads them all and jumps to the exec address automatically.

Path 2 — DriveWire 4

DriveWire 4 runs on your modern computer and connects to the CoCo via a USB-to-serial adapter. It presents virtual disk drives to the CoCo over the serial link.

  1. Install DriveWire 4 on your computer and place your .bin file in a virtual disk directory.
  2. Connect the USB-serial adapter between computer and CoCo's serial port (or bit-banger port).
  3. At the BASIC prompt: 
    CLOADM"GUESS.BIN"
    BASIC fetches the file over the serial link exactly as if it were a physical disk drive.

Path 3 — Cassette Tape

If you have a CoCo but none of the modern storage options, the audio output of your computer can stand in for a tape deck. The decb tool from the toolshed package converts a .bin file to a .wav audio file; the CoCo reads it over its cassette input. 

$ decb copy -t guess.bin guess.wav
# play guess.wav to the CoCo's cassette input while running:
# CLOADM (CoCo waits for tone, then loads)

Cassette loading is slow (about 1500 baud) and sensitive to audio levels, but it works on any unmodified CoCo with a standard audio cable.

What CLOADM Does

Regardless of the path, CLOADM does the same thing: it reads data blocks from the source, copies each block's payload to the specified load address in RAM, then jumps to the exec address in the end block. For Bare Naked Forth binaries:

Block Load address Contents
Cursor init$00502-byte cursor variable (0)
Kernel $1000DOCOL, CFAs, primitives, START
Application$2000Compiled Forth thread
Exec $1225 (approx)Jumps to START in kernel

START clears the screen, initialises the stacks, and begins executing the application thread at $2000. BASIC is gone — the CoCo is running your Forth program.

Staying in XRoar

XRoar is a full-speed, cycle-accurate CoCo 2 emulator. For development purposes it is indistinguishable from real hardware. The -run flag loads and executes the binary exactly as CLOADM would: 

$ xroar -machine coco2bus \
        -bas ~/.xroar/roms/bas12.rom \
        -extbas ~/.xroar/roms/extbas11.rom \
        -run getting-started/examples/12/guess.bin

The Full Build Workflow

Here is the complete path from Forth source to running program, whether you target XRoar or real hardware: 

# 1. Write or edit your Forth source
$ edit getting-started/examples/12/guess.fs

# 2. Build the kernel (only needed if kernel.asm changed)
$ cd forth/kernel && make

# 3. Cross-compile your Forth source
$ python3 forth/tools/fc.py guess.fs \
        --kernel forth/kernel/build/kernel.map \
        --kernel-bin forth/kernel/build/kernel.bin \
        --output guess.bin

# 4a. Run in XRoar
$ xroar -machine coco2bus -run guess.bin

# 4b. Or copy to SD card for CoCoSDC
$ cp guess.bin /Volumes/COCO/

Steps 2 and 3 are fast — a full rebuild takes under a second on any modern computer. The edit-compile-run loop is tight enough that you'll iterate many times per session.

What's Next

You've built a complete Forth system from scratch: a kernel of 6809 machine-code primitives, a cross-compiler, and twelve chapters of progressively more capable programs. The natural next steps from here are:

  • String printing — a PRINT word that takes a counted string from memory, so you don't need to chain hundreds of CHAR/EMIT calls.
  • Multi-digit input — extend DIGIT? into a word that reads a full decimal number from the keyboard.
  • Graphics — the VDG supports semigraphics modes where each character cell becomes a 2×3 block of colour. Stack-based coordinate maths makes this surprisingly clean.
  • An interactive interpreter — the ultimate destination: type Forth words directly at the CoCo prompt and execute them immediately.

Every program in this book was written with the tools introduced in that same book. The kernel is about 500 bytes of assembly; the cross-compiler is about 400 lines of Python. Small tools, built carefully, go a long way.
